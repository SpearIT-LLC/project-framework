# Feature: Virtual Staff Transparency in Meeting Records

**ID:** FEAT-149
**Type:** Feature
**Priority:** Medium
**Created:** 2026-02-22

---

## Summary

Define and implement a transparency standard for meeting records that involve Virtual Staff (AI personas). The goal is full disclosure without minimizing the value — Virtual Staff is a feature, not a limitation to hide.

---

## Problem Statement

Meeting records generated by `/swarm` list AI personas (e.g., Alex, Dan, Sam) as participants alongside human attendees. If these documents are shared with external stakeholders (customers, partners), recipients will question who these people are — and there is currently no standard way to answer that question within the document itself.

**Context:**
First raised after the 2026-02-20 Swarm Kick-off session. The meeting record lists "Alex (Product Owner), Dan (Senior Developer), Sam (Architect)" with no indication these are AI personas.

**Impact:**
Credibility risk if shared externally; inconsistency across meeting records as swarm usage grows.

---

## Options Under Discussion

Three complementary approaches, leaning toward using all three in combination:

**Option 1 — Role Tagging**
Append a VS (Virtual Staff) tag to each AI persona in the participants line:
```
Alex (Product Owner, VS), Dan (Senior Developer, VS), Sam (Architect, VS)
```

**Option 2 — Meeting Method / Type Field**
Add a `Method:` line to the meeting header:
```
Method: AI Swarm (/swarm) — Virtual Staff simulation
```

**Option 3 — Footnote / Disclosure Block**
Add a disclosure at the bottom of the document (distinct from the existing AI-generated footer):
```
Note: Alex, Dan, and Sam are AI personas operating as Virtual Staff — structured AI-assisted analysis simulating expert team perspectives.
```

**Current Lean:** Combine all three. Role tagging makes personas scannable; the method field explains the process; the footnote provides full context for unfamiliar readers.

---

## Requirements

**Must Have:**
- A defined transparency standard documented in the framework
- `/swarm` command updated to apply the standard in generated meeting records
- Standard applied consistently — framework repo and starter template

**Should Have:**
- Guidance on when/how to adapt for external sharing vs. internal records
- Wording that frames Virtual Staff as a practice (positive framing)

**Out of Scope:**
- Retroactively updating existing meeting records
- Changes to how swarm personas are configured

## Alex's Title — Role Clarification

The current swarm command assigns Alex the title "Product Owner" — but the user *is* the product owner. Alex's actual function is facilitation and challenge: asking the hard questions, stress-testing the idea, defining the MVP boundary. "Product Owner" implies organizational authority the user holds.

**Candidates under consideration:**

| Title | Notes |
|-------|-------|
| Consultant | Honest and simple; maps to the product's "AI consultant for consultants" mission |
| Delivery Lead | Retains scope/MVP-boundary focus explicitly in the title |
| Engagement Lead | Neutral, consultancy-flavoured, implies session facilitation |
| Project Advisor | Softer, guidance-oriented |
| Strategic Advisor | Slightly more senior-sounding; appropriate for the challenge role |

**Current lean:** Consultant or Delivery Lead. "Consultant" fits the product identity best.

**Note:** This affects meeting records shared externally — "Product Owner" is a loaded Agile title. A title like "Consultant" reads more naturally to an outside audience and is consistent with the Virtual Staff transparency goal.
